# x402 Protocol — Developer Reference

> Written: 2026-06-14  
> Purpose: Reference for integrating x402 pay-per-request into an autonomous BSC trading agent loop.

---

## Overview

Think of x402 like a parking meter built into every HTTP endpoint. You drive up (make a request), the meter tells you the price (402 response with payment terms), you insert the coin (attach a signed payment header), and the gate opens (server delivers the resource).

**What problem it solves:** APIs and AI inference services today require accounts, subscriptions, or API keys. x402 replaces all of that with a single HTTP round-trip and a stablecoin micropayment. No accounts. No API keys. No billing dashboards. A machine (or AI agent) can pay for exactly what it needs, when it needs it, entirely programmatically.

**Who made it:** Coinbase. Open-sourced at `coinbase/x402`, now maintained as `x402-foundation/x402`.

**Status (June 2026):** Production. 165M+ transactions processed. ~$600M annualized volume. Zero protocol fees.

## Protocol Flow

```
Client                          Server                        Facilitator
  │                               │                               │
  │── GET /paid-endpoint ────────>│                               │
  │                               │                               │
  │<─ 402 Payment Required ───────│                               │
  │   Header: PAYMENT-REQUIRED    │                               │
  │   (base64 JSON, see below)    │                               │
  │                               │                               │
  │ [Client reads payment terms,  │                               │
  │  signs authorization locally] │                               │
  │                               │                               │
  │── GET /paid-endpoint ────────>│                               │
  │   Header: PAYMENT-SIGNATURE   │                               │
  │   (base64 JSON PaymentPayload)│                               │
  │                               │── verify + settle ──────────>│
  │                               │<─ OK ────────────────────────│
  │<─ 200 OK + resource ──────────│                               │
```

### Headers

| Direction       | Header Name         | Content |
|-----------------|---------------------|---------|
| Server → Client | `PAYMENT-REQUIRED`  | Base64-encoded `PaymentRequired` JSON |
| Client → Server | `PAYMENT-SIGNATURE` | Base64-encoded `PaymentPayload` JSON |
| Server → Client | `PAYMENT-RESPONSE`  | Base64-encoded settlement result |

### PaymentRequired Object (what the 402 response carries)

```json
{
  "x402Version": 2,
  "error": "Payment required to access this resource",
  "resource": {
    "url": "https://api.example.com/paid-endpoint",
    "description": "Live BTC/USDT price feed",
    "mimeType": "application/json"
  },
  "accepts": [
    {
      "scheme": "exact",
      "network": "eip155:8453",
      "amount": "1000",
      "asset": "0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913",
      "payTo": "0xReceiverAddress",
      "maxTimeoutSeconds": 300,
      "extra": {}
    }
  ]
}
```

### PaymentPayload Object (what the client sends back)

```json
{
  "x402Version": 2,
  "resource": { "url": "...", "description": "...", "mimeType": "..." },
  "accepted": { /* chosen option from accepts[] */ },
  "payload": {
    "signature": "0x...",
    "authorization": {
      "from": "0xClientWallet",
      "to": "0xReceiverAddress",
      "value": "1000",
      "validAfter": "1700000000",
      "validBefore": "1700000300",
      "nonce": "0xrandom"
    }
  }
}
```

The `authorization` struct is an EIP-3009 `transferWithAuthorization` payload. The client signs it locally — the private key never leaves the client.

### Payment Schemes

| Scheme | Description | Chain Support |
|--------|-------------|---------------|
| `exact` | Fixed price per request | All chains |
| `upto` | Client authorizes max; server charges actual usage | All chains |
| `batch-settlement` | High-frequency micropayments; settle on-chain in batches | EVM only |

## Client SDK

### TypeScript — Install

```bash
# Core + EVM support (for Base, Polygon, Arbitrum, and EVM-compatible chains)
npm install @x402/fetch @x402/core @x402/evm

# If using Axios instead of native fetch
npm install @x402/axios @x402/core @x402/evm

# Add Solana support
npm install @x402/svm

# Add Aptos, Stellar, Algorand support
npm install @x402/aptos @x402/avm @x402/stellar
```

### TypeScript — Minimal Fetch Client

```typescript
import { wrapFetchWithPayment, x402HTTPClient } from "@x402/fetch";
import { x402Client } from "@x402/core/client";
import { ExactEvmScheme } from "@x402/evm/exact/client";
import { privateKeyToAccount } from "viem/accounts";

// 1. Create a signer from a private key (viem-based)
const signer = privateKeyToAccount(process.env.EVM_PRIVATE_KEY as `0x${string}`);

// 2. Build the x402 client and register your chain scheme
const client = new x402Client();
client.register("eip155:*", new ExactEvmScheme(signer));

// 3. Wrap native fetch — payment is handled automatically on 402 responses
const fetchWithPayment = wrapFetchWithPayment(fetch, client);
const httpClient = new x402HTTPClient(client);

// 4. Make requests — if a 402 is returned, the client pays and retries
const response = await fetchWithPayment("https://api.example.com/paid-endpoint");
const result = await httpClient.processResponse(response);
console.log(result.body);
```

### TypeScript — Axios Client

```typescript
import axios from "axios";
import { wrapAxiosWithPayment } from "@x402/axios";
import { x402Client } from "@x402/core/client";
import { ExactEvmScheme } from "@x402/evm/exact/client";
import { privateKeyToAccount } from "viem/accounts";

const signer = privateKeyToAccount(process.env.EVM_PRIVATE_KEY as `0x${string}`);
const client = new x402Client();
client.register("eip155:*", new ExactEvmScheme(signer));

const axiosWithPayment = wrapAxiosWithPayment(axios.create(), client);

const { data } = await axiosWithPayment.get("https://api.example.com/paid-endpoint");
```

### TypeScript — Multi-Chain Client (EVM + Solana)

```typescript
import { ExactEvmScheme } from "@x402/evm/exact/client";
import { ExactSvmScheme } from "@x402/svm/exact/client";
import { createKeyPairSignerFromBytes } from "@solana/kit";
import { base58 } from "@scure/base";
import { privateKeyToAccount } from "viem/accounts";

const evmSigner = privateKeyToAccount(process.env.EVM_PRIVATE_KEY as `0x${string}`);
const svmSigner = await createKeyPairSignerFromBytes(
  base58.decode(process.env.SVM_PRIVATE_KEY!)
);

const client = new x402Client();
client.register("eip155:*", new ExactEvmScheme(evmSigner));
client.register("solana:*", new ExactSvmScheme(svmSigner));
```

## Server / Facilitator Side

The server has two jobs:

1. **Protect routes:** Return HTTP 402 with payment terms when an unauthenticated request arrives.
2. **Verify payments:** When a `PAYMENT-SIGNATURE` header is present, forward it to a facilitator for on-chain verification, then serve the resource if valid.

Servers don't verify blockchain state themselves — they delegate that to a facilitator (a trusted third-party service or self-hosted contract).

### Express.js Server Example

```bash
npm install @x402/express @x402/core @x402/evm
```

```typescript
import express from "express";
import { paymentMiddleware } from "@x402/express";

const app = express();

app.use(paymentMiddleware({
  facilitatorUrl: "https://x402.org/facilitator", // testnet facilitator
  routes: {
    "GET /api/price-feed": {
      accepts: [
        {
          scheme: "exact",
          network: "eip155:8453",   // Base mainnet
          amount: "1000",           // 1000 wei of USDC (= $0.001)
          asset: "0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913", // USDC on Base
          payTo: process.env.RECEIVER_ADDRESS!,
          maxTimeoutSeconds: 300,
        }
      ],
      description: "Live price feed per request",
    }
  }
}));

app.get("/api/price-feed", (req, res) => {
  res.json({ btcPrice: 67000 });
});

app.listen(3000);
```

### Protocol-Level Support (x402-foundation)

The open protocol uses CAIP-2 identifiers and supports:
- Any EVM chain via `eip155:<chainId>` (e.g., `eip155:56` for BSC)
- Solana, TON, Algorand, Stellar, Aptos, Hedera at protocol level

Chains with pre-configured defaults in `@x402/evm` include: Base, Polygon, Arbitrum, Monad, Mezo, MegaETH, XDC Network, and others. The list grows via community contributions.

## BSC Compatibility

### Short Answer

**BSC is NOT in the official Coinbase CDP facilitator.** However, **a third-party facilitator (AEON) has deployed x402 support on BNB Chain.**

### Technical Barrier

The default `ExactEvmScheme` in `@x402/evm` relies on **EIP-3009** (`transferWithAuthorization`) for gasless token transfers. The major BSC stablecoins — notably **USDT on BSC (BEP-20)** — do not implement EIP-3009. This means the standard scheme cannot execute gasless payments using BSC USDT.

**USDC on BSC** may support EIP-3009 [UNVERIFIED — check Circle's BSC deployment].

### AEON Facilitator Workaround

AEON (a BNB Chain MVB program graduate) deployed an x402 facilitator on BNB Chain in late 2025. Their approach:
- Supports both **EIP-3009-enabled tokens** and **standard ERC-20 tokens** (without EIP-3009)
- For non-EIP-3009 tokens: the agent wallet must hold **BNB for gas** (no gasless)
- Uses **ERC-8004** for on-chain agent identity receipts

**Status:** Confirmed exists; contract addresses and facilitator endpoint not publicly documented in official sources at time of writing. Contact AEON at https://aeon.xyz or their Telegram.

### Using BSC via `eip155:56` (Protocol-Level)

If you self-host a facilitator or use AEON's, you can target BSC like this:

```typescript
// Register BSC (eip155:56) as your EVM target
client.register("eip155:56", new ExactEvmScheme(signer));

// Your server's PaymentRequired would specify:
{
  "scheme": "exact",
  "network": "eip155:56",
  "amount": "1000000",        // in token's smallest unit
  "asset": "0x<USDC_BSC>",   // USDC contract on BSC [UNVERIFIED address]
  "payTo": "0xYourWallet",
  "maxTimeoutSeconds": 300
}
```

The `eip155:*` wildcard in `client.register` will match BSC if you need multi-chain support.

### Recommendation for This Project

| Option | Effort | Risk |
|--------|--------|------|
| Use Base mainnet (Coinbase CDP) for x402 payments | Low | Low |
| Use AEON facilitator on BSC | Medium | Medium (third-party, unverified docs) |
| Self-host x402 facilitator on BSC | High | High (requires EIP-3009 workaround) |

If this agent must transact on BSC, the pragmatic path is: **use Base for x402 payments** (pay the data/inference endpoints there), while keeping BSC for the actual trading logic. The two chains don't have to be the same.

## Full Working Example (Agent Loop Pattern)

```typescript
import { wrapFetchWithPayment, x402HTTPClient } from "@x402/fetch";
import { x402Client } from "@x402/core/client";
import { ExactEvmScheme } from "@x402/evm/exact/client";
import { privateKeyToAccount } from "viem/accounts";

// ---- Setup (run once) ----
const signer = privateKeyToAccount(process.env.EVM_PRIVATE_KEY as `0x${string}`);
const x402 = new x402Client();
x402.register("eip155:*", new ExactEvmScheme(signer)); // covers Base, Polygon, BSC, etc.

const fetchWithPayment = wrapFetchWithPayment(fetch, x402);
const httpClient = new x402HTTPClient(x402);

// ---- Agent loop ----
async function fetchPriceFeed(endpoint: string): Promise<any> {
  try {
    const response = await fetchWithPayment(endpoint, { method: "GET" });
    const result = await httpClient.processResponse(response);
    return JSON.parse(result.body as string);
  } catch (err: any) {
    if (err.message?.includes("No scheme registered")) {
      throw new Error(`Chain not supported by registered schemes: ${err.message}`);
    }
    if (err.message?.includes("insufficient")) {
      throw new Error("Wallet underfunded — top up USDC or gas");
    }
    throw err;
  }
}

// Usage inside trading loop
while (true) {
  const price = await fetchPriceFeed("https://data.example.com/bnbusdt");
  // ... trading logic ...
  await new Promise(r => setTimeout(r, 1000));
}
```

## Gotchas

1. **EIP-3009 is required for gasless EVM payments.** If your token doesn't have `transferWithAuthorization`, the standard `ExactEvmScheme` will fail. On Base, USDC and EURC work. On BSC, standard USDT does not.

2. **Payments are irreversible.** Once signed and settled on-chain, there are no refunds in the protocol. Design your retry logic carefully — a signature is used up even if the server returns bad data.

3. **Signature replay protection.** The `nonce` + `validBefore` timestamp combination ensures signatures can't be reused. Signatures expire at `validBefore` (default: 5 minutes). Don't cache payment payloads.

4. **Network mismatch errors.** The most common source of failures in testnet→mainnet transition: wrong CAIP-2 IDs or the wrong USDC contract address for the chain.

5. **Agent wallet must be funded.** Your agent's signing key controls the wallet that pays. It needs USDC (or whichever token the server accepts) AND native gas token (ETH on Base, MATIC on Polygon, BNB on BSC) for on-chain operations.

6. **`eip155:*` wildcard.** Registering `eip155:*` means your client will try to pay on ANY EVM chain a server asks for. Be careful — a malicious server could demand payment on an expensive chain. Scope to specific chain IDs for production agents.

7. **`PAYMENT-REQUIRED` is base64.** The header value is `base64(JSON)`. The SDK handles this for you, but if you're debugging raw HTTP, decode the header manually.

8. **Coinbase CDP free tier.** 1,000 transactions/month free, then $0.001/tx. For an agent running in a tight loop, budget accordingly.

9. **BSC facilitator is third-party.** AEON's facilitator is not Coinbase-hosted. Due diligence on reliability, uptime, and trust model is on you.
